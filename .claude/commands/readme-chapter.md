# 章 README の作成

新しい章の `README.md` を以下の構成で作成する。

## 構成

### 1. タイトルと概要（冒頭）

```markdown
# NN タイトル（サブタイトル）

前章の何が問題で、この章で何を解決するかを2〜3行で書く。
変更ファイルも冒頭に明記する。
```

### 2. 起動セクション

minikube / kubectl / frontend の起動手順を順番に書く。章ごとのディレクトリ名を明記する。

### 3. 確認セクション

**具体的な操作手順**（番号付きリスト）と**期待される動作**（箇条書き）をセットで書く。
バックエンドログで確認する方法も記載する。

### 4. ファイル構成セクション

変更があったファイルに `# [変更] 変更内容` を付ける。変更なしは `# NN章と同一` と記載。

```
NN_chapter/
    k8s/             # NN-1章と同一
    backend/
        app/
            module/
                file.py   # [変更] 追加した関数名など
    frontend/
        src/
            hooks/
                useXxx.ts  # [変更] 変更内容
```

### 5. 実装の解説セクション

**前章の問題** → **この章の解決** の流れで書く。ASCII フローで視覚的に示す。

```
### NN章の問題

問題の流れ
    ↓
問題の結果
```

```
### NN+1章の解決

解決の流れ
    ↓
解決の結果
```

変更ファイルごとにコードスニペット（変更箇所のみ）＋インラインコメントで補足。

### 6. 残る制約テーブル

| ケース | 状態 |
|---|---|
| 解決済みのケース | ✅ 〜（この章で解決）|
| 解決済みだが条件付き | ✅ 〜（理由）|
| 未解決のケース | ❌ 〜（理由）|

### 7. クラスターのリセット

```bash
kubectl delete -f k8s/
minikube stop
```

## ルール

- コードスニペットはファイル全体ではなく変更箇所のみ抜粋する
- 変更なしファイルは「NN章と同一」と明記してコピーであることを示す
- 「Docs/」以下の実装を読んで裏付けが取れた情報のみ書く（推測で書かない）
- 期待される動作には「何が出ないか」も含める（例: 「入室しましたが表示されない」）
